# OCL (Online Casino Latino) — How to use Claude with this repo

Astro port of the original static site (`MariaDevs/OCL-test`), wired to the
**Matching Visions Content Engine** at
`C:\Users\chrni\Desktop\matching visions projects\content enigne`.
The engine writes, compliance-checks and delivers articles automatically;
this site pulls them at build time and publishes them under `/noticias/`.

---

## Project layout

| Path | What it is |
|------|-----------|
| `src/fragments/<page>/head.html` + `body.html` | The original site's pages, verbatim. **Edit page content here.** |
| `src/pages/<page>/index.astro` | Thin wrappers wiring fragments into the Shell layout (generated — usually don't touch). |
| `src/layouts/Shell.astro` | Document shell for ported pages (charset, favicon, `/style.css`, `/script.js`). |
| `src/layouts/ArticleLayout.astro` | Full site chrome (header/footer/+18 notice) for engine-generated articles. |
| `src/pages/noticias/index.astro` | **Custom** (`@custom` marker): injects engine articles above the static news grid. |
| `src/pages/noticias/[slug].astro` | One page per delivered engine article. |
| `src/lib/content-engine-loader.ts` | Astro content-collection loader pulling the engine feed at build time. |
| `src/content.config.ts` | Defines the `articles` collection. |
| `public/` | `style.css`, `script.js`, `favicon.svg`, `logos/`, `robots.txt`. |
| `scripts/convert-from-static.mjs` | One-shot converter from the original static repo. Skips pages whose `.astro` contains `@custom`. |

Sitemap is generated by `@astrojs/sitemap` at `/sitemap-index.xml` (robots.txt
already points there) — engine articles are included automatically.


## Editing the site (rules for Claude)

1. **Page copy/content** → edit `src/fragments/<page>/body.html` (or `head.html`
   for SEO meta / JSON-LD). These are raw HTML, rendered byte-for-byte.
2. **Site-wide chrome for article pages** → `src/layouts/ArticleLayout.astro`.
   Note: ported pages each carry their own header/footer inside their body
   fragment (exactly like the original site) — a nav change across all pages
   means a search-replace across `src/fragments/**/body.html`.
3. If you customize a generated `index.astro`, add a `// @custom` comment so
   the converter never overwrites it.
4. **Compliance**: this is a gambling-affiliate site (+18). Keep the
   responsible-gambling notices, `rel="sponsored nofollow"` on affiliate links,
   and never add "I tested / first-hand" claims to content. Run the
   `igaming-compliance` skill before publishing significant new content.
5. The Mundial 2026 section is fenced with `<!-- WC2026:START/END -->` comments
   in the fragments so it can be stripped after the World Cup.
   **Post-tournament deletion checklist** — these standalone WC2026 files live
   OUTSIDE the comment-fence mechanism and must be deleted manually when the
   fenced blocks are stripped (otherwise the page stays live and indexable):
   - `src/pages/mundial/estadisticas-mundial-2026/` (whole directory)
   - `public/wc2026-data.js`
   - `public/wc2026-stats.js`

## Content Engine integration (how articles arrive)

**Flow:** engine planner (hourly cron) → research → draft → humanize → SEO →
**compliance gate** → delivered to this site's feed → webhook fires → Vercel
deploy hook rebuilds this site → loader pulls the feed → article pages appear
under `/noticias/<slug>/` and on the `/noticias/` index. No humans in the loop.

### One-time setup

1. **Engine dashboard → Sites → Add site** — name "Online Casino Latino".
   ⚠️ The onboarding script currently accepts markets `uk dk fi se no de global`
   and languages `en da fi sv no de`. **Spanish (`es`) and a LATAM market are
   not supported yet** — use `--market global` and ask for `es` to be added to
   the engine's `MARKETS`/`LANGUAGES` lists + compliance ruleset before
   enabling the schedule, otherwise articles come out in the wrong language.
2. **Dashboard → Keys** — create a site-scoped API key with `articles:read`.
3. **Vercel project for this repo** — create a **Deploy Hook** (Settings →
   Git → Deploy Hooks), and set it as the site's `delivery_webhook_url` in the
   engine (plus a `delivery_webhook_secret`, min 16 chars). Vercel deploy hooks
   don't verify signatures — that's fine, the hook only triggers a rebuild and
   the actual content is pulled with the API key.
4. **Env vars** (locally in `.env`, and in Vercel → Environment Variables):

   ```
   CE_API_URL=https://<engine-deployment>/v1
   CE_API_KEY=<site-scoped key>
   ```

   Without them the build still succeeds — `/noticias/` just shows only the
   static news. Delivery mode for this site is **feed** (not WordPress).

### Onboarding / topping up topics

Run Claude **in the engine repo** (`content enigne` folder) and use the prompt
in its `HOW-TO-USE-CLAUDE.md`. Example onboarding command the engine expects:

```bash
node --env-file=.env.local scripts/onboard-site.mjs \
  --name "Online Casino Latino" --market global --language es \
  --mode feed --per-week 5 --topics ocl-topics.txt
```

Topic seeds should be Spanish, LATAM search-intent angles (bonos, métodos de
pago locales como Mercado Pago/Oxxo/PSE/Yape, regulación por país, guías de
juegos) — opinion/analysis only, never first-hand "I tested" claims.

Status / calendar check (in the engine repo):

```bash
node --env-file=.env.local scripts/show-calendar.mjs 7
```

### Re-syncing from the original static repo

If the old static site changes and you want to re-import:

```bash
git clone --depth 1 https://github.com/MariaDevs/OCL-test.git "$env:TEMP\OCL-test"
npm run sync:static    # converts, preserves @custom pages
```

## Offers feed (casino bonus + CTA links from the central dashboard)

Casino cards in `src/fragments/**/body.html` carry stable markers:

- `data-offer="<slug>" data-offer-field="bonus"` on the leaf element holding
  the bonus text (`.bonus-amount` / `.top3-bonus`)
- `data-offer="<slug>" data-offer-field="cta"` on the outbound affiliate
  anchor (`.cta-btn` / `.top3-cta`)

Slugs are the folder names under `src/fragments/casinos/` (e.g. `betsson`,
`22bet`, `ultra-casino`). After `astro build`, `scripts/apply-offers.mjs`
fetches `GET {CE_API_URL}/offers` (same `CE_API_URL` / `CE_API_KEY` env vars
and Bearer auth as the article feed) and rewrites **dist/ only**: bonus marker
text is replaced with the offer's `bonusText`, CTA `href`s with `ctaUrl`.
`rel`, `target` and classes are never touched, and source fragments are never
modified by the apply step.

Fallback: if the env vars are missing or the feed is unreachable, the step
logs and exits 0 — the site ships with the hardcoded text from the fragments.
`status: "paused"` offers are not removed in v1; the build log prints a
warning listing the pages still showing that casino.

Notes for editing:

- The markers live in the source fragments, so they survive normal content
  edits. If you add a **new** casino card, add the two markers yourself or
  re-run `node scripts/add-offer-markers.mjs` (idempotent — it only tags
  untagged cards it can attribute with certainty).
- Keep the bonus element a leaf (text only) — the apply step replaces its
  entire inner content.
- `node scripts/test-apply-offers.mjs` runs an end-to-end check against a
  local mock feed and rebuilds dist afterwards.
